# AGENTS.md Review

Review agent instruction files as operational guidance, not prose documentation. The goal is to determine whether a coding agent will make fewer avoidable mistakes after reading the file.

## Workflow

1. Locate the repository root and instruction file. Prefer `AGENTS.md`; also accept `CLAUDE.md`, `.cursorrules`, `.cursor/rules`, or a user-specified file.
2. If the file is meant to be generated by `tiny-agents-md`, first check whether the deterministic loop is stable:

   ```bash
   tiny-agents-md init . --dry-run
   ```

   If it is not stable, mention that the generated file is out of sync before reviewing subjective usefulness.
3. Run the local hard checker when available:

   ```bash
   tiny-agents-md doctor . --json
   ```

   If the console script is unavailable inside the `tiny-agents-md` repo, use:

   ```bash
   python -m tiny_agents_md doctor . --json
   ```

   Treat this output as evidence for factual issues. If neither command is available, manually inspect the same evidence from repo files.
4. Read only the files needed to verify claims:
   - Agent file under review
   - `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, and `Makefile` when present
   - README lightly, only to detect copied prose or missing high-signal workflow information
   - Referenced paths, only to confirm existence and generated/build status
5. Evaluate in this order:
   - **Factual correctness:** commands are declared, paths exist, package manager matches lockfiles.
   - **Actionability:** commands are copy-pasteable, rules name concrete files/paths/actions.
   - **Minimality:** content is short and avoids README-style project summaries.
   - **Boundaries:** generated/vendor/build outputs, dependency changes, config changes, and verification expectations are clear when relevant.
   - **Coverage:** detectable test/lint/build/install commands and key source/test paths are present when they exist.

## Output

Keep the review concise and evidence-based:

```text
Verdict: excellent | good | needs work | misleading
Score: N/100

Hard Issues:
- ...

Usefulness Gaps:
- ...

Suggested Minimal Patch:
...
```

Omit empty sections. Prefer a small patch over rewriting the file from scratch. Do not reward generic phrases such as "follow best practices", "write clean code", or long project overviews unless they contain concrete, repo-specific instructions.
